Privacy boundary

The hackathon story succeeds only if this separation is obvious in the product, code and pitch.

Private FieldSeal application

Actual work records stay here

  • Customer names and addresses
  • Workers, roles and credentials
  • Photos, measurements and files
  • Plans, findings and report content
  • Encrypted package manifest
  • Recipient permissions and expiry
AES-256-GCM package encryption and ordinary application access control remain FieldSeal responsibilities.
salted 32-byte commitment only

Midnight network

Proof state lives here

  • Receipt identifier
  • Document commitment
  • Anchor transaction state
  • Valid / revoked status
  • Independent verification result
No report text, personal data, filenames, measurements or customer details belong on-chain.

Must demonstrate

  • Private report remains readable only with access
  • Same report creates the same verifiable commitment
  • Changed report fails verification
  • Access can expire or be revoked
  • Midnight receipt remains independently checkable

Useful if time permits

  • Recipient QR or share link
  • One public synthetic verification page
  • US role labels: contractor, field professional, reviewer / AHJ
  • Jurisdiction and source-reference placeholders
  • Mobile evidence capture polish

Do not build this weekend

  • Billing, subscriptions or marketplaces
  • Automated legal conclusions
  • Nationwide code or permit engine
  • Public worker reputation scores
  • Production customer integrations
  • Mainnet migration unless required

Quality & safety

Definition of done

  1. A manager creates a realistic work order and assigns a team.
  2. A field professional records evidence and submits completed work.
  3. A reviewer approves the exact version used to issue the report.
  4. FieldSeal encrypts the report and anchors only its salted commitment.
  5. A recipient receives limited access and verifies the report against Midnight.
  6. The team can explain in one sentence what stays private and what goes on-chain.
